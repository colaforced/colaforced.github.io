我给 Aerospike 数据库提了一个 31 倍性能优化的 Patch

一次从 O(N) 到 O(1) 的旅程：如何在不改客户端的情况下，让一个数据库内核操作快 31 倍。

背景

Aerospike 是一个高性能分布式 KV 数据库，广泛用于广告、推荐、风控等对延迟极其敏感的场景。它的核心代码是纯 C 实现，以极致性能著称。

我在生产环境中遇到一个性能问题：一个看起来很简单的批量查询，把整个集群打崩了。

问题

业务需求很简单——批量取 10000 条酒店记录，同时过滤出 masterHotelId 在一个 9000 元素白名单中的记录。用 SQL 表达就是：

定位

第一反应：是不是用法不对？

尝试了各种 workaround：

方案 结果
A=1 OR A=2 OR ... 9000 个条件 序列化体积爆炸，更慢
按 key 分片缩短 list 小流量可用，大流量仍崩
用 Map 代替 List 立即数 Map 没有有序标记，一样慢
Bitmap 值域 0~133M，单请求 bitmap >16MB

全部失败。问题不在用法，在内核。

真相大白：客户端发送的常量列表在 wire 上是普通 msgpack array，缺少 Aerospike CDT 的有序标记（ext header + flags）。服务端检查 list_is_ordered() 时看不到这个标记，于是对每条记录都做 O(N) 线性扫描。

即使客户端预排序了列表也没用——“有序”是 CDT 存储属性，不是数据的事实属性。服务端无法信任外部数据的排序声明。

方案设计

核心思路

既然不能走有序路径的二分查找，那就造一条更快的路：编译期构建 hash set，求值期 O(1) 查找。

Aerospike 的 Expression 引擎有一个优雅的两阶段架构：

我的优化插入点：

  • 编译阶段：检测到 LIST_GET_ALL_BY_VALUE + EXISTS + 常量列表 ≥ 8 元素 时，从 msgpack 列表构建一个 hash set
  • 求值阶段：如果 hash set 存在，直接 hash lookup 返回结果，跳过整个 CDT 路径

数据结构

  • Hash 函数：cf_wyhash64（代码库已有的高质量非加密 hash）
  • 冲突处理：开放寻址 + 线性探测
  • 负载因子：50%（capacity = count × 2）
  • 去重：插入时 hash + memcmp 去重

关键设计决策

为什么不改 CDT 层？ CDT 层是共享的通用数据结构引擎，改动影响面太大。而 Expression 编译阶段是一个天然的优化窗口——它已经有类似的先例（GeoJSON 在编译期预计算 region/cellid）。

为什么不用 server 已有的有序数据结构？ CDT ordered-list 需要特定的 msgpack 元数据格式，从客户端强制注入这些元数据既脆弱又侵入。hash set 更通用，且 O(1) 优于 O(log N)。

对其他场景的影响？ 零。不满足检测条件时 fast_set_p == NULL，唯一开销是 eval_call 中一次指针判空（~1 CPU cycle，分支预测 always-not-taken）。

可视化

过程中的一个 Bug

初始实现中，我将检测条件写成了 AS_CDT_OP_LIST_GET_BY_VALUE（op=20）。部署后发现优化没有生效——延迟仍然线性增长。

通过在编译路径插入诊断日志，发现 Java 客户端的 ListExp.getByValue() 实际发送的是 AS_CDT_OP_LIST_GET_ALL_BY_VALUE（op=22）。前者是”获取第一个匹配”，后者是”获取所有匹配”——EXISTS 语义需要后者。

一个字符的差异，让优化从”完全不工作”变成”31 倍提升”。这类 bug 没有 unit test 能提前覆盖到，只有端到端的 benchmark 才能暴露。

复杂度对比

维度 优化前 优化后
编译（per batch） O(1) O(M) 建 hash set
求值（per key） O(M) 线性扫描 O(1) hash lookup
10000 key × 9000 list 9000 万次 msgpack 比较 9000 次 hash insert + 10000 次 hash lookup
额外内存 0 ~432 KB
其他路径影响 — 1 次 NULL 判断

总结

这个优化的核心 insight 是：数据库查询优化器的经典技巧（compile-time hash build）可以应用到 NoSQL 的 expression 引擎中。

PostgreSQL 在 Hash Semi-Join 中做完全相同的事——对常量子查询建内存 hash table，然后 probe。区别只是 Aerospike 的 “子查询” 是 expression 中的常量列表，”probe” 是每条记录的 bin 值。

技术栈：C (gnu99) / msgpack / wyhash64 / open addressing hash table / Aerospike Expression Engine

改动规模：1 个文件，+170 行，零依赖新增，零接口变更

效果：31x 延迟降低，9000 万次比较降至 1 万次 hash lookup

完整 patch 和 benchmark 代码已提交至 aerospike-server 的 perf/exp-fast-set-list-in 分支。
