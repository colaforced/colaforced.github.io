Guava RateLimiter 源码：令牌桶不是你想的那样

Created2026-01-10|Updated2026-06-30
|Post Views:

背景

网上大多数介绍文章说，RateLimiter 使用了令牌桶算法，且支持预热模式。这个说法没有错，但一旦试图理解”预热”的实现机制，就会遇到一个核心矛盾：

桶满了，生成令牌速度最慢，但桶里有令牌，不应该直接放行吗？怎么就能实现冷启动了？

带着这个疑问读源码，发现 Guava 的实现和直译的令牌桶有本质区别。

直译实现 vs Guava 实现

常见的令牌桶直译实现是：起一个定时任务以固定速率往桶里放令牌，应用取到令牌就放行，取不到就等。

Guava 的实现完全不同——基于惰性填充的令牌桶。核心逻辑是：

  1. 不起任何定时任务
  2. 每次 acquire() 时，根据距上次的时间差，计算应补充多少令牌
  3. 关键不再是”能不能拿到令牌”，而是”拿到令牌需要等多久”

即使桶满了，在预热模式下依然需要等待才能放行。这就解释了冷启动为什么能生效。

核心理解：用存储令牌数计算等待时间，从而控制放行速度。预热时令牌越多等待越久，不预热时有令牌就不等。

实现机制

惰性填充

不起任务单独算令牌，而是在外部要令牌时计算时间差，补充对应数量的令牌。简单高效。

等待时间计算

分两种模式：

SmoothBursty（突发模式）

桶里令牌够 → 返回 0，直接放行。不够，假设还缺 N 块 → 等待 N × 令牌生成间隔 后放行。

SmoothWarmingUp（预热模式）

存储令牌数和等待时间的关系由下面这张图定义：

X 轴是存储令牌数 storedPermits，Y 轴是生成一个令牌所需的时间 throttling。对曲线做积分，面积就是等待时间。

当存储令牌多（冷状态）时，每个令牌对应的等待时间更长 → 放行更慢 → 实现预热。随着令牌被消耗，曲线左移，等待时间逐渐缩短 → 系统进入稳态。

storedPermitsToWaitTime

这个函数的职责是：你要 N 个令牌，我在曲线上从右往左算这 N 个令牌对应的面积（积分），就是等待时间。

  • 不预热：速率恒定，曲线是平行 X 轴的直线，缺口令牌数 × 固定生成时间 即可
  • 预热：对分段函数求面积，梯形部分用梯形面积公式

预设参数

用户能输入的参数只有两个：

  • warmUpPeriod：预热时间
  • permitsPerSecond：稳态吞吐速度

作者在源码中限定了一些魔数：

  • 冷却因子 coldFactor = 3（cold interval = 3 × stable interval）
  • 稳态时间 = 0.5 × warmUpPeriod
  • 冷却速度 = warmUpPeriod / maxPermits

这些是工程经验值。作者注释里说：3 用起来不错，就不动了。配合稳态时间比例，曲线表现合理就这么用下来了。

其他问题

Q：为什么冷却速度用 warmUpPeriod / maxPermits？

只是需要一个固定速度来控制令牌自然恢复。预热周期除以总令牌数，量级合理就行。

Q：里面用了不少 synchronized，JDK21 虚拟线程会有问题吗？

会。synchronized 会 pin 住虚拟线程的载体线程，高并发下可能导致载体线程池耗尽。如果在虚拟线程环境使用，需要注意这个限制，或考虑用 ReentrantLock 替代。
